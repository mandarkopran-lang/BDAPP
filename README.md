# Machine Breakdown Management System

A production-ready, multi-tenant **Flutter + Firebase** app for reporting and
managing machine breakdowns across multiple companies, with OTP login and
role-based access control (Admin / Manager / Supervisor / User).

---

## 1. Tech Stack

| Layer          | Technology |
|-----------------|------------|
| Frontend        | Flutter 3.22+ (Dart 3.3+) |
| Auth            | Firebase Phone Authentication (OTP) |
| Database        | Cloud Firestore |
| File storage    | Firebase Storage |
| State mgmt      | Provider |
| Routing         | go_router |
| CI/CD           | GitHub Actions (auto APK build) |

## 3. Firestore Data Model

```
users/{uid}
  uid, phoneNumber, name, role (admin|manager|supervisor|user),
  companyId (null until assigned), departmentId, plantId, isActive, createdAt

companies/{companyId}
  name, ownerId, isActive, createdAt, metadata

companies/{companyId}/departments/{deptId}
  companyId, name, description, createdAt

companies/{companyId}/plants/{plantId}
  companyId, name, location, departmentId, createdAt

companies/{companyId}/breakdowns/{breakdownId}
  companyId, plantId, departmentId, title, description,
  status (open|assigned|inProgress|resolved|closed),
  priority (low|medium|high|critical),
  reportedBy, assignedTo, imageUrls[], createdAt, updatedAt, resolvedAt,
  statusHistory[] -> { status, by, at, note }
```

Every tenant-scoped collection lives **under** `companies/{companyId}/...`,
so a Firestore query can never accidentally return another company's data —
this is the core of the multi-tenant isolation, enforced both by app code
(`FirestoreService`) and by `firestore.rules`.

## 4. Firebase Setup Guide (step-by-step)

### 4.1 Create the Firebase project
1. Go to https://console.firebase.google.com → **Add project**.
2. Name it (e.g. `breakdown-management-prod`). Enable/disable Analytics as you prefer.

### 4.2 Enable required services
- **Authentication** → Sign-in method → enable **Phone**.
- **Firestore Database** → Create database → Start in **production mode** (rules are provided below).
- **Storage** → Get started → production mode.

### 4.3 Register your Android app
1. Project Settings → **Add app** → Android.
2. Package name: `com.example.breakdown_management_app` (or change it — see §4.6).
3. Download `google-services.json` and place it at `android/app/google-services.json` (replacing the placeholder).

### 4.4 Generate `firebase_options.dart` automatically
```bash
dart pub global activate flutterfire_cli
flutterfire login
flutterfire configure --project=<your-firebase-project-id>
```
This overwrites `lib/firebase_options.dart` with real values for Android/iOS/Web
and re-downloads `google-services.json` / `GoogleService-Info.plist` for you.
**Do this before your first run** — the file committed in this repo is a
placeholder and will NOT connect to a real project.

### 4.5 Get your SHA-1 / SHA-256 fingerprints (fixes OTP failures)
This is the **#1 cause** of "OTP not working" on Android.

Debug keystore (used automatically by `flutter run`):
```bash
keytool -list -v \
  -keystore ~/.android/debug.keystore \
  -alias androiddebugkey -storepass android -keypass android
```
Release keystore (once you create one — see §6):
```bash
keytool -list -v -keystore <path-to-your-release.keystore> -alias <your-alias>
```
Copy both the SHA-1 and SHA-256 values into:
**Firebase Console → Project Settings → Your Android app → Add fingerprint**.

⚠️ After adding fingerprints, **re-download `google-services.json`** — the old
file does not contain the new fingerprints and OTP will still silently fail
until you replace it.

### 4.6 (Optional) Change the Android package name
If you don't want `com.example.breakdown_management_app`, update it in all of:
- `android/app/build.gradle` → `namespace` and `applicationId`
- `android/app/src/main/kotlin/.../MainActivity.kt` package path (move the folder too)
- Firebase Console Android app config → re-download `google-services.json`

### 4.7 Deploy security rules & indexes
```bash
npm install -g firebase-tools
firebase login
firebase use --add            # select your project
firebase deploy --only firestore:rules,firestore:indexes,storage
```

### 4.8 Add test phone numbers (recommended during development)
Firebase Console → Authentication → Sign-in method → Phone →
**Phone numbers for testing** → add e.g. `+91 9999999999` with a fixed OTP
`123456`. This bypasses real SMS quota/delivery while you're developing.

### 4.9 Bootstrap your first Admin
There is no seeded admin — by design, the **first person to sign up becomes
a normal "user"** with no company. To make them an Admin who can create a
company:
1. Sign up via OTP in the app once (this creates their `users/{uid}` doc).
2. In Firebase Console → Firestore → `users/{their-uid}` → manually set
   `role: "admin"`.
3. Re-open the app — they'll now see the Admin tab and can create a company
   (which auto-assigns them as that company's admin/owner).

## 6. Building a signed release APK (for Play Store / sideloading)

1. Generate a keystore:
   ```bash
   keytool -genkey -v -keystore ~/release-key.jks -keyalg RSA -keysize 2048 -validity 10000 -alias release
   ```
2. Create `android/key.properties` (already gitignored):
   ```properties
   storePassword=<your-password>
   keyPassword=<your-password>
   keyAlias=release
   storeFile=/absolute/path/to/release-key.jks
   ```
3. Add the release SHA-1/SHA-256 to Firebase Console (§4.5) and re-download
   `google-services.json`.
4. Build:
   ```bash
   flutter build apk --release
   # or per-ABI (smaller APKs):
   flutter build apk --release --split-per-abi
   ```
   Output: `build/app/outputs/flutter-apk/`

## 7. CI/CD — GitHub Actions (auto APK build)

`.github/workflows/flutter-build.yml` builds a release APK on every push to
`main` and uploads it as a workflow artifact; pushing a tag like `v1.0.0`
also attaches the APK to a GitHub Release automatically.

**Required repository secret:**
| Secret name | Value |
|---|---|
| `GOOGLE_SERVICES_JSON` | Full contents of your real `google-services.json` |
| `FIREBASE_OPTIONS_DART` *(optional)* | Full contents of your real `lib/firebase_options.dart` |

Add these under **Repo → Settings → Secrets and variables → Actions → New
repository secret**. Never commit the real files directly if the repo is
public.

## 8. Fixing Common Firebase Issues

### "PERMISSION_DENIED: Missing or insufficient permissions"
Almost always one of:
1. **No `users/{uid}` document exists yet.** `AuthService._ensureUserDocument`
   creates it immediately after OTP sign-in — if you bypassed that flow
   (e.g. manually created a user in the Auth console), create the Firestore
   doc yourself with at least `role` and `companyId` fields.
2. **`companyId` mismatch.** All company-scoped rules check
   `myCompanyId() == companyId` from the resource path — a user with no
   company (or the wrong one) will be denied. Assign them via
   User Management screen or manually in Firestore.
3. **Rules not deployed.** Run `firebase deploy --only firestore:rules`
   after any edit to `firestore.rules` — editing the file locally does
   nothing until deployed.
4. **Querying without the required composite index.** Firestore returns a
   permission-looking error with a console link to auto-create the missing
   index; click it, or predeploy via `firestore.indexes.json` (already
   included here for all queries used by this app).

### "OTP not working" / SMS never arrives
1. Missing SHA-1/SHA-256 fingerprint in Firebase Console (§4.5) — the most
   common cause on Android.
2. Phone sign-in provider not enabled in Authentication settings.
3. Real SMS quota exceeded on the Spark (free) plan — upgrade to Blaze or
   use test phone numbers (§4.8) during development.
4. Using an emulator without Google Play Services — phone auth requires a
   real device or an emulator image with Play Store support.
5. `google-services.json` is stale (downloaded before you added the
   fingerprint) — re-download it.

### "Undefined field" / null errors reading Firestore documents
All models in `lib/core/models/` use defensive `map['field'] ?? default`
parsing, so missing fields won't crash — but if you see this it usually
means a document was created outside the app (e.g. manually in console)
without all expected fields. Compare against the data model in §3 and
backfill the missing fields.

### SHA-1 / signing issues in general
- Debug and release builds use **different keystores** and therefore
  **different SHA-1 values** — you must add both to Firebase.
- If you rotate/regenerate your keystore, old fingerprints remain valid in
  Firebase until you remove them; add the new one, don't just replace.
- `flutter build apk` without `key.properties` configured silently falls
  back to the debug signing config in this project's `build.gradle` — fine
  for testing, but Play Store requires a proper release key.
